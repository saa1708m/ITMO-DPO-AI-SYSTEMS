## Цель работы

Сформировать навыки проектирования высокоуровневой архитектуры ML-платформы и её практической реализации в виде базового docker-compose-стенда с использованием JupyterHub и MinIO.[web:248][web:320]

## Проверяемые результаты обучения

- **Индикатор LC-3.1, средний уровень** — обучающийся проектирует архитектуру с учётом требований к масштабируемости, надёжности и сопровождению, обосновывает выбор технологий.[file:1]
- **Индикатор LC-5.1, средний уровень** — обучающийся конфигурирует многосервисные стеки, настраивает сети и тома для компонентов ML-платформы.[file:1][web:325]

## Формат выполнения

Лабораторная работа выполняется в одном из двух форматов:

- **Обязательный формат** — проектирование архитектуры и разработка docker-compose-стенда для заданного сценария использования ML-платформы.
- **Вариативный формат** — проектирование архитектуры и разработка стенда для самостоятельно выбранного сценария (например, под задачу из курсового проекта) по согласованию с преподавателем.

Базовым и рекомендуемым вариантом является выполнение работы на основе типового сценария, предложенного преподавателем. Это позволяет всем студентам работать в единой предметной области и сосредоточиться на архитектурных решениях и качестве реализации стенда.

## Сценарий для проектирования

Разрабатывается ML-платформа для команды исследователей и инженеров, работающих над проектами машинного обучения. Платформа должна обеспечивать:

- **Среду для разработки** — JupyterHub с возможностью работы нескольких пользователей и запуска персональных notebook-сессий.[web:327]
- **Хранилище артефактов** — MinIO для хранения данных, моделей и других артефактов как S3-совместимое объектное хранилище.[web:320][web:332]
- **Возможность расширения** — добавление новых компонентов (например, MLflow, PostgreSQL, Airflow) в будущем без полной переработки архитектуры.[web:323][web:329]
- **Изоляцию и воспроизводимость** — запуск каждого компонента в отдельном контейнере с фиксированными версиями и явной конфигурацией.[web:325][web:328]

## Постановка задания

Необходимо спроектировать архитектуру ML-платформы и реализовать её базовый стенд. Работа включает следующие этапы:

1. **Проектирование высокоуровневой архитектуры** — определение ключевых компонентов, их взаимодействия и обоснование выбора технологий.
2. **Разработка docker-compose-стенда** — создание конфигурации для запуска JupyterHub и MinIO как взаимодействующих сервисов.[web:248][web:326]
3. **Конфигурирование сети и томов** — настройка сетевого взаимодействия между контейнерами и организация постоянного хранения данных.[web:325]
4. **Тестирование стенда** — проверка работоспособности сервисов и их взаимодействия, в том числе доступа из JupyterHub к MinIO.[web:322]

## Обязательная часть работы

Каждый студент должен:

### 1. Спроектировать высокоуровневую архитектуру ML-платформы

- определить состав компонентов (JupyterHub, MinIO, а также при необходимости дополнительные сервисы);
- обосновать выбор каждого компонента с точки зрения функциональности, масштабируемости и сопровождения;
- описать потоки данных между компонентами;
- построить диаграмму архитектуры (рекомендуется C4-модель, уровень контейнеров).

### 2. Реализовать docker-compose-стенд

- создать файл `docker-compose.yml` для запуска JupyterHub и MinIO;
- настроить переменные окружения для каждого сервиса;
- организовать постоянное хранение данных с использованием Docker volumes;
- настроить общую сеть для взаимодействия контейнеров;
- при необходимости использовать `.env`-файл и healthcheck для повышения управляемости стенда.[web:321][web:325]

### 3. Протестировать стенд

- убедиться, что оба сервиса запускаются и доступны;
- проверить веб-интерфейс JupyterHub;
- проверить веб-интерфейс MinIO;
- проверить, что JupyterHub может обращаться к MinIO, например через `boto3`, `s3fs` или совместимый S3-клиент.[web:322]

### 4. Подготовить документацию и отчёт

- описать процесс запуска и остановки стенда;
- зафиксировать результаты тестирования;
- сформулировать выводы по архитектуре и возможностям её расширения.

## Вариативная часть

Студент может выполнить лабораторную работу для самостоятельно выбранного сценария, если соблюдены все обязательные требования. Например, вместо JupyterHub может использоваться JupyterLab в контейнере, VS Code Server или иная среда, а вместо MinIO — другое объектное хранилище при условии содержательного обоснования выбора.

Для вариативного формата рекомендуется дополнительно:

- обосновать выбор альтернативных компонентов;
- сравнить выбранное решение с базовым сценарием `JupyterHub + MinIO`;
- указать, какие дополнительные компоненты были добавлены и почему.

## Рекомендации по организации работы

Для большинства студентов рекомендуется использовать базовый сценарий `JupyterHub + MinIO`. Это позволяет:

- сосредоточиться на понимании принципов контейнеризации и многосервисной конфигурации;
- получить работающий стенд, который затем можно расширять в следующих модулях;
- обеспечить сопоставимость результатов при проверке.

Выполнение работы с самостоятельным выбором сценария целесообразно разрешать как вариативную или повышенную траекторию для более сильных студентов.

## Требования к отчёту

### 1. Архитектура ML-платформы

В отчёте необходимо представить:

- обоснование выбора компонентов:
  - **JupyterHub** — как многопользовательскую среду для разработки и экспериментов;
  - **MinIO** — как S3-совместимое объектное хранилище для данных и артефактов;
  - дополнительные компоненты (если добавлены) — с обоснованием их необходимости;
- описание потоков данных:
  - как данные попадают в систему;
  - как JupyterHub взаимодействует с MinIO;
  - как сохраняются и извлекаются артефакты;
- диаграмму архитектуры — высокоуровневую диаграмму компонентов и связей между ними (рекомендуется C4 model, уровень контейнеров).[web:310][web:333]

### 2. Реализация docker-compose-стенда

В отчёте необходимо отразить:

- структуру `docker-compose.yml`:
  - описание каждого сервиса (образ, порты, переменные окружения, тома, зависимости);
  - настройку сетей;
  - организацию томов для постоянного хранения;
  - при наличии — healthcheck, restart policy, `.env`-конфигурацию.[web:321][web:325]
- инструкцию по запуску:
  - команды для запуска и остановки стенда;
  - команды или действия для проверки доступности сервисов.

### 3. Тестирование стенда

В разделе тестирования необходимо показать:

- **проверку работы JupyterHub**:
  - доступность интерфейса;
  - вход пользователя и запуск notebook-сессии;
- **проверку работы MinIO**:
  - доступность интерфейса;
  - создание bucket и загрузку файла;
- **проверку взаимодействия**:
  - подключение из JupyterHub к MinIO;
  - чтение и запись данных или артефактов из notebook в MinIO.[web:322][web:333]

### 4. Выводы

Необходимо дать:

- оценку спроектированной архитектуры;
- анализ сильных и слабых сторон текущего стенда;
- предложения по расширению (например, добавление MLflow, PostgreSQL, Airflow, reverse proxy, monitoring).

## Критерии оценивания

### «Отлично»

- Архитектура спроектирована полно и обоснованно.
- Выбор компонентов аргументирован с учётом требований к масштабируемости, надёжности и сопровождению.
- Docker Compose-стенд полностью работоспособен, сервисы корректно взаимодействуют.
- Диаграмма архитектуры наглядна и соответствует описанию.
- Отчёт структурирован и содержит все необходимые разделы.

### «Хорошо»

- Архитектура в целом спроектирована корректно, но есть отдельные недочёты: не все компоненты обоснованы, диаграмма содержит неточности, стенд работает с небольшими проблемами конфигурации.
- Основные выводы верны, но не всегда достаточно подробно аргументированы.

### «Удовлетворительно»

- Выполнена только часть требований: архитектура описана поверхностно, обоснование выбора компонентов отсутствует или неполное.
- `docker-compose.yml` содержит ошибки, диаграмма отсутствует или слабо связана с описанием.
- Отчёт формален и не демонстрирует достаточного уровня инженерного понимания.

### «Неудовлетворительно»

- Архитектура не спроектирована или не соответствует заданию.
- Docker Compose-стенд не работает.
- Отсутствует один или несколько обязательных артефактов.
- Выводы отсутствуют или не относятся к выполненной работе.
