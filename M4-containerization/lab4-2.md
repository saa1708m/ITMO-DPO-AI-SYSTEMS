## Цель работы

Сформировать навыки сборки многосервисного ML-стенда с использованием Docker Compose: конфигурирование стека сервисов (MLflow, MinIO, PostgreSQL, JupyterHub), настройка сетевого взаимодействия и организации постоянного хранения данных для последующего использования в промышленных MLOps-пайплайнах.[web:360][web:382]

## Проверяемый результат обучения

Индикатор LC-5.1, средний уровень: обучающийся конфигурирует многосервисные стеки и настраивает сети и тома для компонентов ML-платформы.[file:1]

## Формат выполнения

Лабораторная работа выполняется в одном из двух форматов:

- обязательный формат — на основе стартового учебного репозитория преподавателя, содержащего шаблон ML-стенда с базовой конфигурацией Docker Compose;
- вариативный формат — в собственном репозитории студента с самостоятельной сборкой ML-стека или на основе собственной заготовки.

Базовым и рекомендуемым вариантом является выполнение работы на основе стартового репозитория. Это позволяет всем студентам работать в единой структуре проекта и сосредоточиться на настройке взаимодействия сервисов и организации хранения данных, а не на поиске и подборе совместимых версий компонентов.[web:360]

## Стартовый репозиторий

Преподаватель предоставляет стартовый учебный репозиторий, содержащий минимально необходимую структуру проекта:

- базовый (неполный) файл `docker-compose.yml` с определением одного–двух сервисов;
- файлы конфигурации для сервисов (при необходимости);
- переменные окружения (`.env`‑заготовка);
- инструкцию по запуску и тестированию стенда.[web:360][web:383]

## Постановка задания

Дан учебный репозиторий, содержащий заготовку ML-стенда. Необходимо доработать и настроить многосервисный стек, обеспечивающий работу ключевых компонентов ML-платформы.

Работа включает следующие этапы:

1. **Конфигурирование стека сервисов** — добавление в `docker-compose.yml` всех необходимых сервисов: MLflow (трекинг экспериментов и реестр моделей), MinIO (S3‑совместимое хранилище артефактов), PostgreSQL (база данных для метаданных MLflow), JupyterHub (среда для разработки).[web:360][web:382]
2. **Настройка сетевого взаимодействия** — организация сети для обеспечения взаимодействия между контейнерами.
3. **Организация постоянного хранения** — настройка Docker volumes для сохранения данных и артефактов между перезапусками контейнеров.[web:383][web:389]
4. **Тестирование стенда** — проверка работоспособности всех сервисов и их взаимодействия.
5. **Документирование** — краткое описание структуры стека, принятых решений по конфигурации и результатов тестирования.

## Обязательная часть работы

Каждый студент должен:

1. Получить копию стартового репозитория (fork или клонирование).
2. Доработать `docker-compose.yml`, добавив и настроив следующие сервисы:
   - **MLflow** — с использованием PostgreSQL в качестве бэкенда для метаданных и MinIO в качестве хранилища артефактов;
   - **MinIO** — S3‑совместимое объектное хранилище;
   - **PostgreSQL** — база данных для хранения метаданных экспериментов MLflow;
   - **JupyterHub** — многопользовательская среда для разработки с возможностью доступа к данным из MinIO и, при необходимости, к MLflow.[web:360][web:382]
3. Настроить сети и тома:
   - создать пользовательскую сеть для взаимодействия всех сервисов;
   - настроить именованные тома для сохранения данных PostgreSQL, артефактов MinIO и директории с артефактами/логами MLflow.[web:383][web:389]
4. Настроить переменные окружения через `.env`‑файл:
   - задать учётные данные для MinIO (access key и secret key);
   - настроить строку подключения PostgreSQL для MLflow (`MLFLOW_BACKEND_STORE_URI`, например `postgresql://user:pass@postgres/mlflow`);[web:380][web:382]
   - указать S3‑endpoint и `MLFLOW_ARTIFACT_URI` для хранения артефактов MLflow в MinIO (например, `s3://mlflow/`).[web:380][web:383]
5. Запустить стенд и проверить работоспособность:
   - все сервисы запускаются без ошибок;
   - JupyterHub доступен по адресу `http://localhost:8000` (или иному, указанному в конфигурации);
   - MLflow Tracking UI доступен по адресу `http://localhost:5000` (или иному адресу);
   - MinIO Console доступен, например, по адресу `http://localhost:9001`;
   - из ноутбука Jupyter можно подключиться к MinIO и MLflow (создать эксперимент и записать артефакты).[web:360][web:383][web:389]
6. Подготовить краткое описание структуры стека и принятых решений по конфигурации.

## Вариативная часть

Студент может выполнить лабораторную работу в собственном репозитории с самостоятельной сборкой ML-стенда, если соблюдены все обязательные требования к заданию. Такой формат допускается как альтернативный, но требует сохранения всей требуемой функциональности: работающий стек из четырёх сервисов, сетевое взаимодействие, постоянное хранение данных.

Для студентов, выполняющих работу в вариативном формате, рекомендуется дополнительно:

- обосновать выбор версий образов и параметров конфигурации;
- сравнить свою конфигурацию с базовым шаблоном;
- указать, какие элементы были реализованы самостоятельно, а не взяты из шаблона.

## Требуемые артефакты

Обучающийся предоставляет:

- файл `.env` (с примерами значений, без реальных секретов);

## Требования к отчёту

В отчёте следует указать:

- состав стека сервисов и назначение каждого компонента;
- структуру `docker-compose.yml` и обоснование выбранных параметров (образы, порты, переменные окружения, политики рестарта);
- как организованы сети и тома, почему выбран именно такой подход;
- какие переменные окружения настроены и для чего они используются;
- инструкцию по запуску стенда и проверке работоспособности сервисов;
- результаты тестирования взаимодействия сервисов (например, сценарий «от ноутбука до записи эксперимента в MLflow»);
- выполнялась ли работа на основе стартового шаблона или в собственном репозитории;
- какие трудности возникли при настройке стека и как они были решены.

## Рекомендации по конфигурации

При выполнении работы рекомендуется обратить внимание на следующие аспекты:[web:360][web:388]

| Аспект | Рекомендация |
|--------|--------------|
| Версии образов | Использовать фиксированные теги образов (а не `latest`) для обеспечения воспроизводимости и предсказуемости.[web:365][web:372] |
| Сети | Создавать отдельную пользовательскую сеть для изоляции сервисов стенда и внутреннего взаимодействия. |
| Тома | Использовать именованные тома для данных PostgreSQL и MinIO; для артефактов MLflow — именованный том или привязку к директории хоста. |
| Переменные окружения | Выносить настраиваемые параметры в `.env`‑файл, не хранить секреты в `docker-compose.yml`. |
| Зависимости | Использовать `depends_on` для базового порядка запуска сервисов, понимая, что этого недостаточно для ожидания полной готовности сервисов. |
| Health checks | Добавлять проверки здоровья для сервисов, чтобы гарантировать их готовность перед запуском зависимых компонентов. |
| Логи | Настраивать сбор логов и, при необходимости, отдельные драйверы логирования для диагностики проблем. |
| Порты | Публиковать на хост только необходимые порты, избегая конфликтов и лишней экспозиции наружу. |

## Критерии оценивания

### «Отлично»

- Docker Compose‑стек полностью работоспособен, все четыре сервиса корректно настроены и взаимодействуют друг с другом.  
- Сети и тома организованы рационально, переменные окружения вынесены в `.env`.  
- Отчёт содержит содержательное обоснование принятых решений и результаты тестирования.  
- Стенд может быть воспроизведён на другой машине по инструкции из отчёта.

### «Хорошо»

- Стек работает, но есть отдельные недочёты в конфигурации: не все переменные вынесены в `.env`, отсутствуют health checks, настройка томов или сетей не оптимальна.  
- Основной результат достигнут — все сервисы запускаются и взаимодействуют.

### «Удовлетворительно»

- Стек собран и запускается, но есть проблемы с взаимодействием сервисов или организацией хранения данных.  
- Конфигурация содержит избыточные или слабо обоснованные параметры.  
- Отчёт формальный, объяснение структуры стека неполное.

### «Неудовлетворительно»

- Стек не собирается или не запускается.  
- Отсутствует один или несколько обязательных сервисов.  
- Отчёт отсутствует или не соответствует заданию.

## Дополнительные баллы

Дополнительные баллы могут быть предусмотрены за один или несколько из следующих элементов:

- добавление в стек дополнительных сервисов (например, Prometheus + Grafana для мониторинга или Airflow для оркестрации);[web:360]  
- использование профилей Docker Compose для разделения окружений (dev/prod);  
- добавление health checks для всех ключевых сервисов;  
- настройка политик автоматического восстановления контейнеров (`restart`‑policies);  
- использование `docker-compose.override.yml` для локальной разработки;  
- выполнение работы полностью в собственном репозитории с качественным обоснованием конфигурации.
