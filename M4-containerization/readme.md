# Модуль 4. Контейнеризация ML-приложений

## Описание модуля

Модуль посвящён практическому освоению инструментов контейнеризации для задач машинного обучения. Студенты изучают принципы работы Docker, создают оптимизированные образы для обучения и инференса моделей, а также собирают многосервисные стеки с использованием docker-compose для развёртывания компонентов ML‑платформы (MLflow, MinIO, PostgreSQL, JupyterHub).

## Название

Контейнеризация ML-приложений

## Раздел РПД

## Методические указания по выполнению и оцениванию заданий

### Общая характеристика заданий

Тестовые задания предназначены для проведения текущего контроля успеваемости студентов по завершении соответствующего лекционного или лабораторного занятия модуля 4 дисциплины «Промышленная разработка, развёртывание и сопровождение ML‑систем». Задания модуля направлены на формирование практических навыков работы с инструментами контейнеризации в контексте ML‑разработки.

Тест может выполняться:

- в конце лекционного или лабораторного занятия;  
- в качестве самостоятельной домашней работы;  
- в электронной образовательной среде;  
- в рамках подготовки к рубежному контролю и промежуточной аттестации.

Рекомендуемое время выполнения одного теста составляет 10–20 минут.

Каждый тест по практическому занятию содержит 5 заданий, по лекционному — 10 заданий. К каждому из заданий имеется 5 вариантов ответа. В каждом задании имеется только один правильный ответ.

Тесты (по лекционным занятиям 4.1 и 4.2) проверяют:

- знание ключевых понятий Docker (образы, слои, Dockerfile, docker-compose);  
- понимание принципов многоступенчатой сборки и оптимизации образов;  
- понимание архитектуры ML‑стенда на основе docker-compose.

Каждый тест состоит из 10 вопросов с одним правильным ответом.

Лабораторные работы носят прикладной характер:

- в лабораторной работе 4.1 студенты создают Docker‑образ для inference‑сервиса, применяя техники оптимизации;  
- в лабораторной работе 4.2 студенты собирают и конфигурируют многосервисный стек для ML‑платформы с использованием docker-compose.

### Методические указания для студента

Перед выполнением теста и лабораторных работ рекомендуется изучить лекционный материал, обращая особое внимание на:

- базовые понятия Docker: образы, контейнеры, слои, Dockerfile, docker-compose;  
- многоступенчатую сборку (multi‑stage builds) и её преимущества для уменьшения размера финального образа;  
- оптимизацию Docker‑образов: минимизация числа слоёв, использование `.dockerignore`, выбор лёгкого базового образа (например, `python:3.10-slim`);  
- архитектуру ML‑стенда: назначение и взаимодействие сервисов MLflow (трекинг экспериментов и реестр моделей), MinIO (S3‑совместимое хранилище артефактов), PostgreSQL (база данных для метаданных MLflow), JupyterHub (среда для разработки).

При выполнении лабораторных работ:

**Для лабораторной работы 4.1**

- внимательно изучите задание и требования к отчёту;  
- начните с написания базового Dockerfile для простого inference‑сервиса (например, на FastAPI);  
- примените многоступенчатую сборку для разделения этапов установки зависимостей и запуска приложения;  
- оптимизируйте образ: используйте `.dockerignore`, выбирайте подходящий базовый образ, при необходимости объединяйте команды `RUN`;  
- проверьте работоспособность собранного образа (контейнер запускается, сервис отвечает на запросы).

**Для лабораторной работы 4.2**

- изучите файл `docker-compose.yml`, в котором определены сервисы MLflow, MinIO, PostgreSQL, JupyterHub;  
- настройте переменные окружения, тома для сохранения данных и сети для взаимодействия сервисов;  
- поднимите стенд и убедитесь, что все сервисы работают и могут взаимодействовать друг с другом (например, MLflow сохраняет артефакты в MinIO, JupyterHub может обращаться к MLflow и MinIO).

При выполнении кейсов и заданий на анализ архитектуры сфокусируйтесь не только на компонентах, но и на точках их интеграции, а также на потенциальных рисках и ограничениях спроектированной или анализируемой системы.

### Методические указания для преподавателя

Тесты рекомендуется использовать для быстрой проверки знаний после лекций. Они позволяют оценить понимание студентами фундаментальных концепций контейнеризации в контексте ML.

Каждый правильный ответ в тесте оценивается в 1 балл.

Лабораторные работы следует оценивать не только по факту выполнения, но и по качеству и обоснованности принятых инженерных решений (выбор базового образа, применение многоступенчатой сборки, организация томов и сетей и т.п.).

При защите лабораторных работ задавайте уточняющие вопросы, чтобы убедиться в самостоятельности выполнения и осмысленности выбора архитектурных решений.

Максимальная оценка за один тест составляет 5 баллов (практическое занятие) и 10 баллов (лекционное занятие).

Частичный балл за выбор неправильного варианта не начисляется. Если студент указал несколько вариантов ответа при требовании выбрать один, ответ считается неправильным.

## Система оценивания

### Рекомендуемая шкала оценивания для тестов из 5 вопросов

| Количество правильных ответов | Баллы | Рекомендуемая оценка  |
|-------------------------------|-------|------------------------|
| 5                             | 5     | «Отлично»             |
| 4                             | 4     | «Хорошо»              |
| 3                             | 3     | «Удовлетворительно»   |
| 0–2                           | 0–2   | «Неудовлетворительно» |

### Рекомендуемая шкала оценивания для тестов из 10 вопросов

| Количество правильных ответов | Баллы | Рекомендуемая оценка  |
|-------------------------------|-------|------------------------|
| 9–10                          | 5     | «Отлично»             |
| 7–8                           | 4     | «Хорошо»              |
| 5–6                           | 3     | «Удовлетворительно»   |
| 0–4                           | 0–2   | «Неудовлетворительно» |

При использовании рейтинговой системы результат может быть переведён в проценты по формуле  
\( P = \frac{N_{\text{правильных ответов}}}{5} \cdot 100\% \)  
для теста из 5 заданий, либо по аналогичной формуле для теста из 10 заданий.

Минимальным результатом, подтверждающим освоение темы, рекомендуется считать 3 правильных ответа из 5 либо 6 правильных ответов из 10, то есть 60 % верных ответов.

## Рекомендации по проведению контроля

Перед началом тестирования преподавателю следует объявить студентам количество заданий, время выполнения, допустимые материалы и критерии оценивания.

При защите лабораторных работ рекомендуется обращать внимание не только на конечный результат (работающий образ или стенд), но и на понимание студентом принципов работы каждого компонента и их взаимодействия.

После проверки тестов целесообразно провести разбор заданий, вызвавших наибольшие затруднения, с акцентом на типичные ошибки, например:

- непонимание назначения многоступенчатой сборки;  
- путаница между образами и контейнерами;  
- неверная интерпретация роли томов и сетей в docker-compose.

Если значительная часть группы допустила одну и ту же ошибку, соответствующий вопрос рекомендуется повторно разобрать на следующем занятии.

### Рекомендуемые принципы оценивания лабораторных работ

При оценивании лабораторных работ рекомендуется использовать сочетание следующих критериев:

- корректность и полнота реализации задания;  
- работоспособность решения;  
- соответствие результата заявленным этапам и требованиям;  
- качество инженерного оформления (структура Dockerfile, читаемость `docker-compose.yml`, работа с конфигурацией);  
- качество краткого отчёта и устной защиты.

Для базового уровня достаточно, чтобы студент воспроизводимо настроил минимально необходимый образ или стек и мог объяснить его шаги. Для среднего уровня требуется, чтобы студент продемонстрировал осознанную оптимизацию, логичную структуру конфигурации и понимание ограничений решения.
