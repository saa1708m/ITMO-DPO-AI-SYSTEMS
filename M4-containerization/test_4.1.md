# Тест 4.1. Основы контейнеризации для ML: Docker, образы, слои, многоступенчатая сборка, оптимизация размера образов

## Цель

Проверить понимание принципов работы Docker, структуры образов и слоёв, а также умение применять практики оптимизации (многоступенчатая сборка, минимизация размера образа, эффективное использование кэша) при создании контейнеров для ML-приложений.[web:364][web:371]

## Инструкция

Выберите **один** правильный ответ в каждом задании.

## Задания

### 1. Что такое Docker-образ (`image`) с точки зрения его внутренней структуры?

а) Это один исполняемый файл, содержащий всё приложение.  
б) Это набор слоёв (`layers`), каждый из которых соответствует изменениям в файловой системе, внесённым инструкциями Dockerfile.  
в) Это сжатый архив с исходным кодом.  
г) Это виртуальная машина с установленной ОС.

### 2. Для чего в Dockerfile обычно используется инструкция `COPY` вместо `ADD`, особенно в ML-проектах?

а) `COPY` работает более предсказуемо и не имеет дополнительных побочных эффектов, например автоматической распаковки архивов.  
б) `COPY` позволяет копировать файлы только из удалённых репозиториев.  
в) `ADD` не поддерживает копирование каталогов.  
г) `COPY` автоматически сжимает копируемые файлы.[web:363][web:376]

### 3. Какая практика позволяет значительно уменьшить итоговый размер Docker-образа для ML-приложения, использующего компилируемые зависимости?

а) Использование тега `latest` для базового образа.  
б) Применение многоступенчатой (`multi-stage`) сборки, где на финальном этапе копируются только необходимые артефакты, а инструменты сборки остаются в промежуточном образе.  
в) Установка всех зависимостей через `pip install --no-cache-dir`.  
г) Добавление всех файлов проекта в один слой.[web:371]

### 4. Почему порядок инструкций в Dockerfile критически важен для эффективного использования кэша при пересборке образа?

а) Порядок влияет на права доступа к файлам.  
б) Docker кэширует результат выполнения каждой инструкции, поэтому часто изменяемые файлы лучше копировать ближе к концу Dockerfile, чтобы переиспользовать кэш на более тяжёлых шагах, например при установке зависимостей.  
в) Порядок влияет на скорость работы контейнера после запуска.  
г) Порядок определяет, какой порт будет открыт.[web:364]

### 5. Какой из перечисленных подходов **не** является способом уменьшения размера Docker-образа для ML-модели?

а) Использование минимального базового образа, например `python:3.10-slim` вместо `python:3.10`.  
б) Удаление временных файлов и кэша пакетного менеджера в том же слое, где они были созданы.  
в) Установка всех зависимостей через `RUN pip install -r requirements.txt` без опции `--no-cache-dir`.  
г) Использование многоступенчатой сборки для исключения компиляторов и заголовочных файлов из финального образа.[web:348][web:352]

### 6. Для ML-сервиса, который загружает обученную модель весом 1 ГБ, какая рекомендация по организации слоёв является наиболее эффективной с точки зрения времени сборки и размера образа?

а) Добавить файл модели в последний слой сразу после копирования исходного кода.  
б) Скопировать файл модели отдельной инструкцией `COPY` как можно раньше, чтобы при изменении кода не пересобирать слой с моделью, если модель меняется редко.  
в) Поместить модель во внешний том (`volume`) или объектное хранилище, а не в образ, чтобы не увеличивать его размер и не усложнять пересборку.  
г) Использовать команду `RUN wget` для загрузки модели при каждом запуске контейнера.

### 7. Что такое многоступенчатая сборка (`multi-stage build`) в Docker?

а) Использование нескольких базовых образов в одном контейнере одновременно.  
б) Возможность описать несколько этапов сборки в одном Dockerfile, при этом на финальный этап копируются только необходимые артефакты, что позволяет создавать минимальные финальные образы.  
в) Сборка образа на нескольких хостах одновременно.  
г) Пошаговое выполнение инструкций с подтверждением пользователя.[web:371]

### 8. Для чего в Dockerfile для Python-приложения рекомендуется сначала копировать `requirements.txt`, выполнять `pip install`, а уже потом копировать остальной код?

а) Это делает сборку более безопасной.  
б) Это позволяет использовать кэш Docker: если `requirements.txt` не изменился, то слой с установкой зависимостей будет взят из кэша, что ускоряет сборку.  
в) Это автоматически уменьшает размер образа.  
г) Это обязательно по синтаксису Dockerfile.[web:364]

### 9. В чём заключается потенциальная опасность использования тега `latest` для базового образа или собственного образа в CI/CD-пайплайне?

а) Это всегда увеличивает время сборки.  
б) Это может привести к непредсказуемому поведению, поскольку содержимое тега может измениться без явного обновления версии, нарушая воспроизводимость.  
в) Это запрещено политикой безопасности Docker Hub.  
г) Это делает образ несовместимым с Kubernetes.[web:365][web:372]

### 10. Какая дополнительная мера, помимо многоступенчатой сборки, помогает уменьшить размер финального образа Python-проекта с большим количеством зависимостей?

а) Использование `pip install --no-deps` для отключения зависимостей.  
б) Использование `pip install --no-cache-dir` и удаление `~/.cache/pip` в том же слое, чтобы кэш не попадал в финальный образ.  
в) Упаковка всех зависимостей в один файл `.zip`.  
г) Запуск контейнера без использования сетевых интерфейсов.[web:348][web:352]
