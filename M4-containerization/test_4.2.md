# Тест 4.2. ML-стенд на docker-compose: стек сервисов MLflow, MinIO, PostgreSQL, JupyterHub, конфигурирование сети и томов

## Цель

Проверить понимание архитектуры и конфигурации ML-стенда на основе Docker Compose, включающего сервисы для трекинга экспериментов (MLflow), хранения артефактов (MinIO), метаданных (PostgreSQL) и интерактивной разработки (JupyterHub), а также умение настраивать сети и тома для обеспечения их взаимодействия и сохранения состояния.

## Инструкция

Выберите **один** правильный ответ в каждом задании.

## Задания

### 1. Для решения задачи организации единого стенда для ML-команды, включающего хранение моделей, артефактов экспериментов и метаданных, какой набор сервисов в docker-compose является наиболее типичным и функциональным?

а) MySQL + Prometheus + Grafana  
б) MLflow + MinIO + PostgreSQL  
в) Hadoop + Spark + Hive  
г) Kafka + Flink + Zookeeper  
д) Jenkins + Nexus + GitLab

### 2. Какова основная роль MinIO в составе описанного ML-стенда?

а) Хранение метаданных экспериментов (параметров, метрик).  
б) Хранение артефактов моделей, датасетов и других файлов, совместимое с S3 API.  
в) Выполнение ETL-преобразований данных.  
г) Обеспечение веб-интерфейса для запуска Jupyter-ноутбуков.  
д) Мониторинг производительности сервисов.

### 3. Для чего в стеке сервисов обычно используется PostgreSQL?

а) Как основное хранилище для JupyterHub.  
б) Для хранения метаданных и состояния экспериментов, используемых MLflow.  
в) Для хранения данных модели в виде BLOB-объектов.  
г) Как балансировщик нагрузки между контейнерами.  
д) Для кэширования результатов инференса модели.

### 4. При конфигурировании сети в docker-compose для описанного ML-стенда, какая практика является наиболее правильной для обеспечения изоляции и обмена данными между сервисами?

а) Использовать сеть `host` для всех сервисов.  
б) Не определять сеть, позволяя Compose создать сеть по умолчанию.  
в) Определить кастомную сеть `bridge` для внутреннего взаимодействия сервисов и пробросить наружу только порты JupyterHub и MLflow UI.  
г) Использовать отдельную сеть для каждого сервиса.  
д) Отключить все сетевые интерфейсы для безопасности.

### 5. Какая конфигурация томов (volumes) является предпочтительной для сервиса MinIO в составе ML-стенда, чтобы обеспечить сохранность данных при перезапуске контейнера?

а) Использовать `tmpfs` для хранения данных в оперативной памяти.  
б) Использовать bind mount с указанием конкретной директории на хосте для хранения данных.  
в) Не использовать тома, хранить данные внутри контейнера.  
г) Использовать named volume, создаваемый и управляемый Docker.  
д) Использовать volume типа `none`.

### 6. В `docker-compose.yml` для сервиса MLflow необходимо задать параметры, указывающие на адрес бэкенда для хранения метаданных и артефактов. Как они будут выглядеть для стека MLflow + PostgreSQL + MinIO (S3‑совместимое хранилище)?

а) `MLFLOW_BACKEND_STORE_URI=sqlite:///mlflow.db` и `MLFLOW_ARTIFACT_URI=./artifacts`  
б) `MLFLOW_BACKEND_STORE_URI=postgresql://user:pass@postgres/mlflow` и `MLFLOW_ARTIFACT_URI=s3://mlflow/`  
в) `MLFLOW_BACKEND_STORE_URI=mysql://user:pass@db/mlflow` и `MLFLOW_ARTIFACT_URI=file:///mlflow/`  
г) `MLFLOW_BACKEND_STORE_URI=mongodb://db/mlflow` и `MLFLOW_ARTIFACT_URI=hdfs://minio/mlflow`  
д) `MLFLOW_BACKEND_STORE_URI=postgresql://db/mlflow` и `MLFLOW_ARTIFACT_URI=minio://bucket`

### 7. Для чего в docker-compose стеке сервисов используется директива `depends_on`?

а) Для определения порядка запуска контейнеров, чтобы зависимые сервисы (например, MLflow) запускались после своих зависимостей (PostgreSQL, MinIO).  
б) Для копирования файлов из одного контейнера в другой.  
в) Для указания портов, которые должны быть опубликованы.  
г) Для монтирования томов.  
д) Для настройки переменных окружения.

### 8. Какая проблема может возникнуть при использовании JupyterHub в составе docker-compose стенда без специальной настройки томов?

а) JupyterHub не сможет подключиться к интернету.  
б) Ноутбуки и файлы пользователей не будут сохранены при перезапуске контейнера JupyterHub, если не использовать том для директории с рабочими данными.  
в) JupyterHub будет потреблять слишком много памяти.  
г) JupyterHub не сможет запустить Python-код.  
д) JupyterHub будет конфликтовать по портам с MLflow.

### 9. Какой тип тома в Docker Compose обычно используется для сохранения состояния базы данных PostgreSQL между перезапусками контейнера?

а) bind mount  
б) named volume  
в) volume с типом `cached`  
г) volume с типом `delegated`  
д) `tmpfs` mount.

### 10. При конфигурировании docker-compose стенда для ML-команды, почему важно использовать `docker-compose.override.yml` или явно разделять базовую и переопределяемую конфигурацию?

а) Для того, чтобы иметь возможность использовать разные версии Docker.  
б) Для упрощения процесса сборки образов.  
в) Для разделения настроек для разработки и продакшена, позволяя переопределять параметры (например, порты, переменные окружения, пути к томам) без изменения основного файла.  
г) Для автоматической регистрации сервисов в реестре контейнеров.  
д) Для одновременного запуска нескольких экземпляров одного сервиса.
