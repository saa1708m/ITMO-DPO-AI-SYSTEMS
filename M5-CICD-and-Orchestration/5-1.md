# Тест 5.1. CI/CD для ML

## Цель

Проверить понимание основных принципов CI/CD в ML-проектах, структуры типичного конвейера, различий между DevOps и MLOps, а также назначения линтеров, тестов, сборки и публикации Docker-образов.

## Инструкция

Выберите один правильный ответ в каждом задании.

## Задания

1. Какое из утверждений **наиболее точно** отражает отличие MLOps от классического DevOps?<br><br>
   а) MLOps используется только для фронтенд-разработки.<br><br>
   б) MLOps учитывает не только код приложения, но и данные, модели, качество предсказаний и цикл переобучения.<br><br>
   в) MLOps не предполагает автоматизацию.<br><br>
   г) MLOps полностью исключает контейнеризацию.<br><br>
   д) MLOps относится только к управлению требованиями.<br><br>

2. Какая последовательность шагов наиболее типична для базового CI-конвейера ML-сервиса?<br><br>
   а) Публикация образа → линтинг → тестирование → сборка.<br><br>
   б) Линтинг → тестирование → сборка образа → публикация артефакта.<br><br>
   в) Обучение модели → A/B-тестирование → аудит безопасности.<br><br>
   г) Оркестрация DAG → сборка образа → настройка RBAC.<br><br>
   д) Мониторинг → анализ дрейфа → деплой.<br><br>

3. Для чего в CI-конвейере обычно используются линтеры (`ruff`, `flake8`)?<br><br>
   а) Для сжатия Docker-образа.<br><br>
   б) Для автоматического улучшения метрик модели.<br><br>
   в) Для проверки качества оформления и синтаксической корректности кода.<br><br>
   г) Для хранения данных обучения.<br><br>
   д) Для публикации артефактов в registry.<br><br>

4. Что является типичным результатом шага `build` в CI-конвейере ML-сервиса?<br><br>
   а) Готовый dashboard в Grafana.<br><br>
   б) Собранный Docker-образ с приложением и зависимостями.<br><br>
   в) Размеченный датасет.<br><br>
   г) Архитектурная диаграмма.<br><br>
   д) DAG в Airflow.<br><br>

5. Какой из инструментов чаще всего используется для модульного тестирования Python-кода в ML-проекте?<br><br>
   а) pytest<br><br>
   б) Grafana<br><br>
   в) MinIO<br><br>
   г) Kubernetes<br><br>
   д) Vault<br><br>

6. Зачем публиковать Docker-образ в registry?<br><br>
   а) Чтобы хранить результаты опросов студентов.<br><br>
   б) Чтобы использовать образ в следующих этапах доставки и развёртывания.<br><br>
   в) Чтобы заменить систему контроля версий.<br><br>
   г) Чтобы запускать линтеры.<br><br>
   д) Чтобы автоматически переобучать модель.<br><br>

7. Какой вариант тегирования Docker-образов является более осмысленным в CI?<br><br>
   а) Использовать только тег `temp`.<br><br>
   б) Не использовать теги вообще.<br><br>
   в) Использовать тег по commit SHA и отдельный тег для основной ветки.<br><br>
   г) Каждый раз случайно генерировать имя вручную.<br><br>
   д) Всегда использовать только `draft`.<br><br>

8. Что из перечисленного лучше всего относится к проверкам качества ML-системы в CI?<br><br>
   а) Проверка схемы данных, unit-тесты и smoke-тест контейнера.<br><br>
   б) Только ручная защита проекта.<br><br>
   в) Только презентация для заказчика.<br><br>
   г) Только запуск ноутбука Jupyter.<br><br>
   д) Только публикация README.<br><br>

9. Какова основная цель этапа Continuous Integration?<br><br>
   а) Отложить проверку кода до экзамена.<br><br>
   б) Регулярно автоматически проверять изменения при их внесении в репозиторий.<br><br>
   в) Исключить тестирование.<br><br>
   г) Заменить разработчиков.<br><br>
   д) Отказаться от ветвления в Git.<br><br>

10. Что является хорошей практикой для CI в ML-проекте?<br><br>
   а) Хранить секреты прямо в файле workflow.<br><br>
   б) Пропускать тесты ради ускорения сборки.<br><br>
   в) Автоматизировать повторяемые проверки и минимизировать ручные операции.<br><br>
   г) Публиковать каждый эксперимент сразу в production.<br><br>
   д) Не фиксировать версии зависимостей.<br><br>
