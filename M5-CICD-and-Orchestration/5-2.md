# Тест 5.2. Оркестрация ML-пайплайнов

## Цель

Проверить понимание принципов оркестрации ML-процессов, DAG-подхода, ролей основных компонентов Apache Airflow и логики автоматического переобучения моделей.

## Инструкция

Выберите один правильный ответ в каждом задании.

## Задания

1. Что такое DAG в контексте Airflow?
   1. Хранилище Docker-образов.
   2. Ориентированный ациклический граф задач и зависимостей между ними.
   3. Формат хранения датасета.
   4. Способ контейнеризации Python-кода.
   5. Модель классификации.

2. Какова основная роль Apache Airflow в ML-проекте?
   1. Создание интерфейсов визуализации.
   2. Оркестрация и планирование последовательностей задач в пайплайне.
   3. Замена Git и систем контроля версий.
   4. Обучение нейронных сетей без кода.
   5. Создание model card.

3. Что обычно понимается под `task` в Airflow?
   1. Весь курс целиком.
   2. Отдельный исполняемый шаг внутри DAG.
   3. Образ Docker Hub.
   4. Ветка Git.
   5. Набор ролей в команде.

4. Что делает scheduler в Airflow?
   1. Выполняет линтинг Python-кода.
   2. Определяет, какие задачи и когда должны быть поставлены на выполнение.
   3. Публикует контейнеры в registry.
   4. Проверяет fairness модели.
   5. Создаёт README.md.

5. Какой сценарий наиболее типичен для auto-retraining?
   1. Модель переобучается вручную только по требованию комиссии.
   2. Модель переобучается при наступлении триггера: новое окно данных, расписание или деградация качества.
   3. Модель переобучается после каждого открытия ноутбука.
   4. Модель никогда не переобучается.
   5. Модель заменяется случайным образом.

6. Какой набор шагов наиболее характерен для DAG переобучения модели?
   1. Загрузка данных → обучение → оценка → принятие решения о публикации/обновлении модели.
   2. Написание презентации → проведение опроса → публикация статьи.
   3. Сборка сайта → публикация баннера → отправка письма.
   4. Создание датасета → удаление модели → остановка сервера.
   5. Сбор логов → дизайн интерфейса → регистрация домена.

7. Для чего в Airflow может использоваться XCom?
   1. Для обмена небольшими данными между задачами.
   2. Для замены Docker Compose.
   3. Для хранения больших бинарных артефактов моделей.
   4. Для балансировки сетевого трафика.
   5. Для анализа fairness.

8. Что является важным риском при автоматическом retraining?
   1. Полное отсутствие запуска задач.
   2. Продвижение новой модели без проверки качества по целевой метрике.
   3. Использование Python в пайплайне.
   4. Наличие логирования.
   5. Наличие DAG-файла.

9. Какой подход является более корректным при обновлении модели в пайплайне?
   1. Обновлять production-модель всегда, независимо от качества.
   2. Сравнивать новую модель с текущей по заданной метрике и обновлять только при соблюдении критерия.
   3. Выбирать модель случайным образом.
   4. Полностью отказаться от проверки метрик.
   5. Обновлять модель только по названию файла.

10. Что лучше всего показывает зрелый подход к оркестрации ML-пайплайна?
   1. Все действия выполняются вручную в ноутбуке.
   2. Шаги формализованы, зависимости заданы явно, а правила запуска и обновления воспроизводимы.
   3. Все файлы лежат в одной папке без структуры.
   4. Решения не документируются.
   5. Ошибки выполнения не отслеживаются.
