# Тест 6.1. Парадигмы serving и REST API

## Цель

Проверить понимание парадигм serving и использования REST API в ML-проектах, видов inference (batch, online, streaming), API-контрактов, понятий latency/throughput.

## Инструкция

Выберите один правильный ответ в каждом задании.

## Задания

1. Какая парадигма inference лучше всего подходит для сценария, где предсказания нужны пользователю сразу после HTTP-запроса, например при скоринге заявки на сайте?<br><br>
   а) Batch inference.<br><br>
   б) Online inference.<br><br>
   в) Offline training.<br><br>
   г) Streaming inference.<br><br>
   д) Feature engineering pipeline.<br><br>

2. В каком случае наиболее уместен batch inference?<br><br>
   а) Нужно отвечать на REST-запросы пользователей за 100 мс.<br><br>
   б) Нужно обрабатывать поток событий из Kafka в реальном времени.<br><br>
   в) Нужно раз в ночь пересчитать предсказания для 10 миллионов клиентов.<br><br>
   г) Нужно протестировать canary deployment.<br><br>
   д) Нужно уменьшить размер Docker-образа.<br><br>

3. Что лучше всего описывает streaming inference?<br><br>
   а) Модель вызывается только вручную через curl.<br><br>
   б) Модель обучается на всех данных сразу.<br><br>
   в) Модель запускается только внутри Jupyter Notebook.<br><br>
   г) Модель возвращает предсказания только после завершения суточного ETL-процесса.<br><br>
   д) Модель обрабатывает непрерывный поток событий с минимальной задержкой.<br><br>

4. Что является важной частью API-контракта для ML-сервиса?<br><br>
   а) Только название Python-файла с моделью.<br><br>
   б) Только версия операционной системы.<br><br>
   в) Формат входных данных, формат ответа, коды ошибок и версия API.<br><br>
   г) Только размер обучающего датасета.<br><br>
   д) Только метрика качества модели на train set.<br><br>

5. Какой пример REST API endpoint наиболее корректен для online inference сервиса?<br><br>
   а) GET /train_model_now.<br><br>
   б) POST /predict.<br><br>
   в) DELETE /model.<br><br>
   г) PUT /dataset.<br><br>
   д) PATCH /requirements.txt.<br><br>

6. Почему для inference чаще используют POST /predict, а не GET /predict?<br><br>
   а) Потому что POST позволяет передавать структурированное тело запроса, например JSON с признаками.<br><br>
   б) Потому что GET-запросы не поддерживаются браузерами.<br><br>
   в) Потому что POST всегда быстрее GET.<br><br>
   г) Потому что GET нельзя использовать в REST API.<br><br>
   д) Потому что POST автоматически обучает модель.<br><br>

7. Какое утверждение про latency и throughput верно?<br><br>
   а) Latency — это количество запросов в секунду, throughput — время ответа на один запрос.<br><br>
   б) Latency и throughput означают одно и то же.<br><br>
   в) Latency — это время обработки одного запроса, throughput — количество запросов, обрабатываемых за единицу времени.<br><br>
   г) Latency относится только к batch inference.<br><br>
   д) Throughput невозможно измерить для REST API.<br><br>

8. Команда разворачивает модель через MLflow Serving. В чём основное преимущество такого подхода по сравнению с ручной реализацией REST API с нуля?<br><br>
   а) MLflow Serving автоматически улучшает качество модели.<br><br>
   б) MLflow Serving предоставляет стандартный способ поднять inference endpoint для зарегистрированной модели.<br><br>
   в) MLflow Serving заменяет необходимость логировать метрики.<br><br>
   г) MLflow Serving работает только с TensorFlow-моделями.<br><br>
   д) MLflow Serving нужен только для batch inference.<br><br>

9. Какой сценарий лучше всего описывает canary deployment модели?<br><br>
   а) Новая модель сразу заменяет старую для 100% пользователей.<br><br>
   б) Новая модель тестируется только локально в ноутбуке.<br><br>
   в) Модель обучается на случайной подвыборке данных.<br><br>
   г) Небольшая доля трафика направляется на новую модель, остальная — на стабильную версию.<br><br>
   д) Все запросы дублируются в batch pipeline без ответа пользователю.<br><br>

10. В чём ключевое отличие A/B-теста модели от canary deployment?<br><br>
   а) A/B-тест используется только для batch inference.<br><br>
   б) Canary deployment всегда требует TensorFlow Serving, а A/B — MLflow Serving.<br><br>
   в) A/B-тест направлен на сравнение вариантов по бизнес- или продуктовым метрикам, а canary чаще используется для безопасного постепенного выката.<br><br>
   г) A/B-тест невозможен для ML-моделей.<br><br>
   д) Canary deployment не требует мониторинга.<br><br>
