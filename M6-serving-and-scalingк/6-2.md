# Тест 6.2. Масштабирование и устойчивость сервисов

## Цель

Проверить понимание принципов масштабирования и устойчивости сервисов, Kubernetes, HPA, балансировки нагрузки, canary deployment, A/B-тестирования, shadow mode.

## Инструкция

Выберите один правильный ответ в каждом задании.

## Задания

1. Какой Kubernetes-механизм обычно используют для автоматического увеличения или уменьшения количества реплик inference-сервиса в зависимости от нагрузки?<br><br>
   а) ConfigMap.<br><br>
   б) Horizontal Pod Autoscaler.<br><br>
   в) PersistentVolumeClaim.<br><br>
   г) Init Container.<br><br>
   д) Namespace.<br><br>

2. ML-сервис с моделью развёрнут в Kubernetes. При росте числа запросов latency резко увеличивается, CPU у pod’ов стабильно выше 85%, а очередь запросов растёт. Какое архитектурное изменение наиболее обосновано в первую очередь?<br><br>
   а) Уменьшить количество реплик сервиса, чтобы снизить конкуренцию за ресурсы.<br><br>
   б) Настроить HPA на масштабирование по CPU или custom-метрике нагрузки.<br><br>
   в) Отключить readiness probe, чтобы Kubernetes быстрее направлял трафик на pod’ы.<br><br>
   г) Перенести все логи в stdout без изменения scaling-политики.<br><br>
   д) Удалить Service и обращаться к pod’ам напрямую по IP.<br><br>

3. Для чего в Kubernetes чаще всего используется readiness probe у inference-сервиса?<br><br>
   а) Чтобы определить, готов ли pod принимать пользовательский трафик.<br><br>
   б) Чтобы автоматически обучать модель после старта pod’а.<br><br>
   в) Чтобы хранить веса модели в etcd.<br><br>
   г) Чтобы увеличить размер Docker-образа.<br><br>
   д) Чтобы заменить Service Mesh.<br><br>

4. Что лучше всего описывает canary deployment ML-модели?<br><br>
   а) Новая модель сразу получает 100% production-трафика без промежуточного контроля.<br><br>
   б) Новая модель получает небольшую долю production-трафика, после чего её поведение сравнивается со стабильной версией.<br><br>
   в) Новая модель запускается только локально на ноутбуке разработчика.<br><br>
   г) Новая модель используется только для offline batch inference и никогда не получает online-запросы.<br><br>
   д) Новая модель обучается на синтетических данных без деплоя.<br><br>

5. Какое утверждение наиболее точно отражает отличие A/B-тестирования от canary deployment?<br><br>
   а) A/B-тестирование обычно используется для сравнения вариантов по продуктовым или бизнес-метрикам, а canary — для безопасного постепенного выката новой версии.<br><br>
   б) A/B-тестирование невозможно применять к ML-моделям.<br><br>
   в) Canary deployment всегда требует наличия двух разных Kubernetes-кластеров.<br><br>
   г) A/B-тестирование не требует сбора метрик.<br><br>
   д) Canary deployment применяется только для batch inference.<br><br>

6. Что такое shadow mode при деплое ML-модели?<br><br>
   а) Новая модель получает 100% трафика, а старая полностью отключается.<br><br>
   б) Модель запускается только ночью для batch-предсказаний.<br><br>
   в) Модель работает без логирования входов и выходов.<br><br>
   г) Kubernetes скрывает pod’ы от балансировщика нагрузки.<br><br>
   д) Новая модель получает копию реального трафика, но её ответы не возвращаются пользователю.<br><br>

7. Inference-сервис на TensorFlow Serving периодически отвечает с высокой задержкой. Метрики показывают, что CPU используется умеренно, но GPU загружен на 100%, а batch-запросы ждут обработки. Какое объяснение узкого места наиболее вероятно?<br><br>
   а) Узкое место обязательно находится в DNS Kubernetes.<br><br>
   б) Узкое место связано только с количеством Namespace’ов.<br><br>
   в) Узкое место невозможно выявить по метрикам загрузки ресурсов.<br><br>
   г) Узкое место вызвано тем, что используется REST API, а не YAML.<br><br>
   д) Узкое место находится в GPU-инференсе или стратегии batching.<br><br>

8. Какая связка Kubernetes-объектов типично используется, чтобы запустить несколько реплик inference-сервиса и распределять между ними трафик внутри кластера?<br><br>
   а) Secret и ConfigMap.<br><br>
   б) Namespace и PersistentVolume.<br><br>
   в) Job и CronJob.<br><br>
   г) Ingress и Init Container без Deployment.<br><br>
   д) Deployment и Service.<br><br>

9. MLflow Serving сервис развёрнут в Kubernetes. Команда хочет сравнить две версии модели: v1 и v2, направляя 90% пользователей на v1 и 10% на v2, при этом собирая latency, error rate и бизнес-метрику конверсии. Какая стратегия деплоя описана наиболее точно?<br><br>
   а) Полный blue-green switch без метрик.<br><br>
   б) Только batch inference.<br><br>
   в) Shadow mode, потому что ответы обеих моделей возвращаются пользователям одновременно.<br><br>
   г) Canary deployment или A/B-тест с разделением трафика между версиями.<br><br>
   д) Rolling update без возможности сравнения версий.<br><br>

10. После выката новой версии модели error rate вырос с 0.2% до 5%, p95 latency увеличилась в 3 раза, а бизнес-метрика ухудшилась. Какая реакция команды наиболее корректна с точки зрения устойчивости сервиса?<br><br>
   а) Продолжить раскатку до 100%, чтобы быстрее собрать больше данных.<br><br>
   б) Отключить мониторинг, чтобы снизить нагрузку на сервис.<br><br>
   в) Удалить старую стабильную модель, чтобы исключить расхождения между версиями.<br><br>
   г) Откатить новую версию или остановить увеличение трафика на canary, затем проанализировать логи, метрики и различия в поведении модели.<br><br>
   д) Увеличить timeout клиентов без анализа причины деградации.<br><br>
