# Лабораторная работа 6.1. REST API для модели

## Цель работы

Сформировать практические навыки развёртывания ML-модели в виде inference-сервиса и оценки его эксплуатационных характеристик.

В ходе выполнения работы студент должен:

* реализовать HTTP API для получения предсказаний ML-модели;
* подготовить сервис к локальному запуску и контейнеризации;
реализовать базовую обработку ошибок и валидацию входных данных;
* добавить служебные endpoints для проверки состояния сервиса;
* провести нагрузочное тестирование;
* измерить latency, throughput и error rate;
* выявить возможные bottlenecks;
* предложить способы оптимизации inference-сервиса.

## Проверяемый результат обучения

Индикатор LC-5.3, базовый уровень: развёртывает REST API с моделью (FastAPI/Flask); выполняет базовое тестирование.

## Постановка задания

### Исходные условия

Студенту предоставляется один из вариантов:

**Вариант А — готовая модель**

Преподаватель предоставляет:

* файл обученной модели, например model.pkl, model.joblib, model.onnx, model.pt;
* описание входных признаков;
* примеры входных JSON-запросов;
* тестовый набор данных.

**Вариант Б — собственная модель**

Студент использует ML-модель, обученную ранее в рамках курса, например:

* модель классификации;
* модель регрессии;
* модель ранжирования;
* NLP-модель;
* простую CV-модель.

Модель должна быть сохранена в виде артефакта и загружаться inference-сервисом при старте приложения.

### Основная задача
Необходимо разработать inference-сервис на FastAPI или Flask, который предоставляет HTTP API для получения предсказаний модели.

Сервис должен поддерживать минимум следующие endpoints:

| Endpoint |	Метод |	Назначение |
|----------|--------|------------|
| /health |	GET |	Проверка работоспособности сервиса |
| /metadata |	GET |	Информация о модели и сервисе |
| /predict |	POST |	Получение предсказания для одного объекта |
| /predict_batch |	POST |	Получение предсказаний для набора объектов |

### Требования к endpoint /health
Endpoint должен возвращать информацию о состоянии сервиса.

Пример ответа:

```
{
  "status": "ok",
  "model_loaded": true,
  "service": "credit-scoring-inference",
  "version": "1.0.0"
}
```
Если модель не загружена, endpoint должен возвращать соответствующий статус и код ошибки или диагностическое сообщение.

### Требования к endpoint /metadata
Endpoint должен возвращать сведения о модели.

Пример ответа:

```
{
  "model_name": "credit_scoring_model",
  "model_version": "1.0.0",
  "framework": "scikit-learn",
  "task_type": "binary_classification",
  "features": [
    "age",
    "income",
    "loan_amount",
    "employment_years"
  ],
  "output": "probability_default"
}
```
Желательно указать:

* название модели;
* версию модели;
* тип задачи;
* список входных признаков;
* формат входа;
* формат выхода;
* дату подготовки модели;
* используемый ML-фреймворк.
### Требования к endpoint /predict
Endpoint должен принимать JSON с признаками одного объекта и возвращать результат предсказания.

Пример запроса:

```
{
  "age": 35,
  "income": 85000,
  "loan_amount": 300000,
  "employment_years": 7
}
```
Пример ответа:

```
{
  "prediction": 0,
  "probability": 0.18,
  "model_version": "1.0.0"
}
```
Для задачи регрессии формат ответа может быть таким:

```
{
  "prediction": 253000.75,
  "model_version": "1.0.0"
}
```
### Требования к endpoint /predict_batch
Endpoint должен принимать список объектов.

Пример запроса:

```
{
  "items": [
    {
      "age": 35,
      "income": 85000,
      "loan_amount": 300000,
      "employment_years": 7
    },
    {
      "age": 52,
      "income": 42000,
      "loan_amount": 150000,
      "employment_years": 3
    }
  ]
}
```
Пример ответа:

```
{
  "predictions": [
    {
      "prediction": 0,
      "probability": 0.18
    },
    {
      "prediction": 1,
      "probability": 0.76
    }
  ],
  "model_version": "1.0.0"
}
```
### Валидация входных данных
Сервис должен проверять:

* наличие обязательных признаков;
* корректность типов данных;
* допустимые диапазоны значений;
* отсутствие лишних или некорректных полей, если это предусмотрено схемой;
* пустые запросы;
* некорректный JSON.

Для FastAPI рекомендуется использовать Pydantic.

Для Flask можно использовать:

* marshmallow;
* pydantic;
* ручную валидацию;
* flask-restx / flask-smorest.

Пример ошибки:

```
{
  "error": "ValidationError",
  "message": "Field 'income' must be greater than 0"
}
```
### Обработка ошибок
Сервис должен корректно обрабатывать:

* неверный формат запроса;
* отсутствие обязательных признаков;
* ошибку загрузки модели;
* ошибку выполнения предсказания;
* обращение к неизвестному endpoint;
* batch-запрос с пустым списком объектов.

Не допускается возвращать пользователю необработанные stack traces.

### Логирование
Сервис должен логировать:

* старт приложения;
* факт загрузки модели;
* обращения к /predict и /predict_batch;
* ошибки валидации;
* ошибки инференса;
* время выполнения предсказания.

Минимальный пример логируемых полей:

* timestamp
* request_id
* endpoint
* status_code
* latency_ms
* model_version

Дополнительно можно логировать:

* размер batch;
* client IP;
* тип ошибки;
* используемый threshold;
* количество предсказаний.

### Контейнеризация
Необходимо подготовить Dockerfile, позволяющий собрать и запустить сервис в контейнере.

Минимальные требования:

* сервис запускается внутри контейнера;
* порт проброшен наружу;
* модель доступна внутри контейнера;
* зависимости устанавливаются воспроизводимо;
* команда запуска указана явно.

Пример запуска:

```
docker build -t ml-inference-service .
docker run -p 8000:8000 ml-inference-service
```
Для FastAPI запуск может быть выполнен через uvicorn:

```
uvicorn app.main:app --host 0.0.0.0 --port 8000
```
Для Flask — через gunicorn:

```
gunicorn app:app --bind 0.0.0.0:8000
```
### Нагрузочное тестирование
Необходимо провести нагрузочное тестирование endpoint /predict и, при наличии, /predict_batch.

Можно использовать один из инструментов:

* Locust;
* k6;
* JMeter;
* ApacheBench;
* wrk;
* Vegeta;
* собственный Python-скрипт.

Минимально необходимо провести не менее трёх сценариев нагрузки.

Пример сценариев:

| Сценарий |	Описание |
|----------|-----------|
| Низкая нагрузка |	1–5 виртуальных пользователей |
| Средняя нагрузка |	10–50 виртуальных пользователей |
| Высокая нагрузка |	100+ виртуальных пользователей или до появления деградации |
Для каждого сценария нужно измерить:

* общее число запросов;
* RPS / throughput;
* среднее время ответа;
* median latency;
* p95 latency;
* p99 latency;
* максимальное время ответа;
* error rate;
* CPU/RAM, если возможно.

### Анализ результатов нагрузочного тестирования
В отчёте необходимо:

1. Представить таблицу с результатами.
2. Построить график зависимости latency от нагрузки.
3. Построить график зависимости throughput от числа пользователей или RPS.
4. Определить точку деградации сервиса.
5. Указать возможные причины деградации.
6. Предложить способы оптимизации.

Примеры возможных оптимизаций:

* загрузка модели один раз при старте приложения;
* использование batch inference;
* настройка количества workers;
* переход с Flask dev server на Gunicorn;
* использование Uvicorn/Gunicorn workers;
* кэширование;
* оптимизация preprocessing;
* использование ONNX Runtime;
* уменьшение размера модели;
* горизонтальное масштабирование;
* асинхронная обработка, если она действительно оправдана;
* вынесение тяжёлых задач в очередь.
