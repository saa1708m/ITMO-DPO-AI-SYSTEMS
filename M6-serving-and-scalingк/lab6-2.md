# Лабораторная работа 6.2. Kubernetes и canary deployment

## «Развёртывание ML inference-сервиса в Kubernetes: HPA, canary- и A/B-деплой, роллбэк»

---

## 1. Цель работы

Сформировать практические навыки промышленного развёртывания ML inference-сервисов в Kubernetes, настройки масштабирования и управления жизненным циклом версий модели.

В ходе работы студент должен:

- развернуть ML inference-сервис в Kubernetes;
- описать Kubernetes-манифесты для приложения;
- настроить `Deployment`, `Service`, `ConfigMap`, `Secret`;
- задать `readinessProbe` и `livenessProbe`;
- настроить resource requests и limits;
- включить Horizontal Pod Autoscaler;
- провести нагрузочное тестирование и проверить автоскейлинг;
- реализовать canary-деплой новой версии модели;
- реализовать A/B-деплой двух версий модели;
- выполнить rollback на предыдущую стабильную версию;
- зафиксировать результаты экспериментов в отчёте.

## 3. Постановка задания

### 3.1. Исходные условия

Студенту предоставляется или используется ранее разработанный ML inference-сервис.

Сервис должен иметь:

- HTTP API;
- endpoint `/health`;
- endpoint `/predict`;
- endpoint `/metadata`;
- Dockerfile;
- опубликованный Docker-образ или возможность собрать его локально.

В лабораторной работе используются две версии сервиса:

- **версия v1** — стабильная версия модели;
- **версия v2** — новая версия модели.

Версии могут отличаться:

- моделью;
- threshold;
- текстом ответа;
- версией в `/metadata`;
- latency;
- качеством предсказаний;
- наличием искусственно внесённой ошибки для проверки rollback.

Пример `/metadata` для версии v1:

```json
{
  "model_name": "credit_scoring_model",
  "model_version": "1.0.0",
  "service_version": "v1"
}
```

Пример `/metadata` для версии v2:

```json
{
  "model_name": "credit_scoring_model",
  "model_version": "2.0.0",
  "service_version": "v2"
}
```

Работа может выполняться в одном из окружений:

- Minikube;
- Kind;
- Docker Desktop Kubernetes;
- k3d;
- удалённый учебный Kubernetes-кластер.

Дополнительно желательно использовать:

- `kubectl`;
- `helm`, если применяется;
- `metrics-server`;
- `hey`, `wrk`, `k6`, `Locust` или другой инструмент нагрузки.

### 3.3. Этап 1. Подготовка Docker-образов

Необходимо подготовить два Docker-образа inference-сервиса:

```text
ml-service:v1
ml-service:v2
```

или, если используется registry:

```text
registry.example.com/ml-service:v1
registry.example.com/ml-service:v2
```

Для каждой версии сервис должен возвращать свою версию через `/metadata`.

Студент должен:

1. Собрать Docker-образы.
2. Проверить локальный запуск контейнеров.
3. Убедиться, что endpoints `/health`, `/metadata`, `/predict` работают.
4. Загрузить образы в registry или сделать их доступными для локального Kubernetes-кластера.

---

### 3.4. Этап 2. Базовое развёртывание сервиса в Kubernetes

Необходимо создать Kubernetes-манифесты для версии v1:

- `Deployment`;
- `Service`;
- `ConfigMap`;
- `Secret`, если в приложении есть параметры, имитирующие секреты;
- при необходимости `Ingress`.

Минимальный набор:

```text
k8s/
  deployment-v1.yaml
  service.yaml
  configmap.yaml
  secret.yaml
```

### Deployment должен содержать:

- имя приложения;
- образ версии v1;
- минимум 2 реплики;
- labels;
- selectors;
- container port;
- env-переменные из `ConfigMap`;
- env-переменные из `Secret`, если используется;
- resource requests;
- resource limits;
- `readinessProbe`;
- `livenessProbe`.

Пример требований к ресурсам:

```yaml
resources:
  requests:
    cpu: "100m"
    memory: "128Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"
```

### Service должен:

- выбирать pods по labels;
- предоставлять доступ к приложению внутри кластера;
- использовать тип `ClusterIP`, `NodePort` или `LoadBalancer` в зависимости от окружения.

---

### 3.5. Этап 3. Проверка доступности и устойчивости

Студент должен проверить:

1. Создание pods:

```bash
kubectl get pods
```

2. Создание service:

```bash
kubectl get svc
```

3. Доступность `/health`.

4. Доступность `/metadata`.

5. Доступность `/predict`.

6. Поведение при удалении одного pod:

```bash
kubectl delete pod <pod-name>
```

После удаления pod Kubernetes должен создать новый pod, а сервис должен остаться доступным.

---

### 3.6. Этап 4. Настройка HPA

Необходимо настроить **Horizontal Pod Autoscaler** для inference-сервиса.

Минимальные требования:

- min replicas: 2;
- max replicas: не менее 5;
- scaling metric: CPU utilization;
- target average utilization: например 50% или 60%.

Пример:

```bash
kubectl autoscale deployment ml-service-v1 \
  --cpu-percent=50 \
  --min=2 \
  --max=5
```

или через YAML-манифест:

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: ml-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: ml-service-v1
  minReplicas: 2
  maxReplicas: 5
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 50
```

---

### 3.7. Этап 5. Проверка автоскейлинга под нагрузкой

Студент должен создать нагрузку на endpoint `/predict`.

Можно использовать:

```bash
hey -z 3m -c 50 http://<service-url>/predict
```

или `k6`, `Locust`, `wrk`.

Необходимо зафиксировать:

- число реплик до нагрузки;
- число реплик во время нагрузки;
- число реплик после снятия нагрузки;
- CPU usage;
- latency;
- error rate;
- время реакции HPA.

Команды для наблюдения:

```bash
kubectl get hpa -w
kubectl get pods -w
kubectl top pods
kubectl describe hpa ml-service-hpa
```

Если HPA не сработал, студент должен проанализировать причину:

- не установлен `metrics-server`;
- слишком низкая нагрузка;
- не заданы resource requests;
- приложение не потребляет CPU;
- нагрузка ушла не на тот service;
- target utilization задан некорректно.

---

### 3.8. Этап 6. Canary-деплой версии v2

Необходимо реализовать canary deployment новой версии модели.

#### Вариант 1. Через два Deployment и общий Service

Создаются:

```text
Deployment v1: replicas = 9
Deployment v2: replicas = 1
Service: selector app=ml-service
```

Обе версии имеют общий label:

```yaml
app: ml-service
```

И разные labels:

```yaml
version: v1
version: v2
```

При таком подходе примерно 10% трафика будет попадать на v2, если количество pod'ов 9 к 1.

Студент должен:

1. Развернуть v2 с малым числом реплик.
2. Проверить, что часть запросов попадает на v2.
3. Увеличить долю v2, например:

```text
v1: 7 replicas
v2: 3 replicas
```

4. Проанализировать ответы `/metadata` или специальное поле `service_version` в ответе `/predict`.
5. Принять решение: продвигать v2 или откатывать.

---

#### Вариант 2. Через Ingress / Service Mesh

Если учебное окружение позволяет, можно использовать:

- NGINX Ingress canary annotations;
- Istio;
- Linkerd;
- Argo Rollouts;
- Flagger.

Например, через NGINX Ingress можно задать вес canary-трафика.

Однако для базовой лабораторной работы достаточно варианта с двумя Deployment и количеством реплик.

---

### 3.9. Этап 7. A/B-деплой

Необходимо реализовать A/B deployment двух версий модели.

В отличие от canary, A/B deployment должен направлять пользователей в разные версии по определённому правилу.

Возможные варианты маршрутизации:

#### Вариант A. Через разные endpoints

- `/predict-a` направляет на v1;
- `/predict-b` направляет на v2.

#### Вариант B. Через разные Services

Создаются два Kubernetes Service:

```text
ml-service-v1
ml-service-v2
```

Далее клиентский скрипт отправляет часть запросов в v1, часть — в v2.

#### Вариант C. Через заголовок

Если используется Ingress или gateway:

```text
X-Experiment-Group: A -> v1
X-Experiment-Group: B -> v2
```

#### Вариант D. Через параметр пользователя

Например:

```text
user_id % 2 == 0 -> v1
user_id % 2 == 1 -> v2
```

Для базовой работы допустим вариант с двумя Services и клиентским скриптом, который распределяет запросы между группами A и B.

Студент должен:

1. Настроить доступ к двум версиям модели.
2. Реализовать или подготовить скрипт отправки запросов.
3. Собрать статистику по группам A и B:
   - количество запросов;
   - latency;
   - error rate;
   - распределение предсказаний;
   - версия модели.
4. Объяснить, чем A/B отличается от canary.

---

### 3.10. Этап 8. Роллбэк

Необходимо продемонстрировать rollback с версии v2 на v1.

Сценарий:

1. Сначала работает v1.
2. Затем выкатывается v2.
3. Обнаруживается проблема:
   - высокий error rate;
   - увеличенная latency;
   - неверный формат ответа;
   - падающий `/health`;
   - неправильная версия модели;
   - имитация ошибки в `/predict`.
4. Выполняется rollback на v1.
5. Проверяется, что сервис снова стабилен.

Возможные способы rollback:

#### Способ 1. Через `kubectl rollout undo`

Если v2 выкатывалась обновлением одного Deployment:

```bash
kubectl rollout history deployment/ml-service
kubectl rollout undo deployment/ml-service
kubectl rollout status deployment/ml-service
```

#### Способ 2. Через масштабирование Deployment

Если используются два Deployment:

```bash
kubectl scale deployment ml-service-v2 --replicas=0
kubectl scale deployment ml-service-v1 --replicas=3
```

#### Способ 3. Через изменение selector или Ingress routing

Можно переключить Service/Ingress обратно на v1.

Студент должен зафиксировать:

- какая проблема была обнаружена;
- какой способ rollback использован;
- сколько времени занял rollback;
- как была проверена успешность отката.
