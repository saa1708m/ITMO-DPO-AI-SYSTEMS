# Тест 7.2. Data drift, concept drift и деградация модели

## Паспорт КИМ

| Параметр | Значение |
|---|---|
| Вид КИМ | Тест |
| Модуль | 7 |
| Тема | Дрейф данных и моделей |
| Компетенция | LC-5 |
| Индикатор | LC-5.4 |
| Уровень | С |
| Время выполнения | 30–40 минут |
| Максимальный балл | 10 |

## Назначение

КИМ предназначен для текущего контроля по теме drift-мониторинга. Тест проверяет различение data drift и concept drift, понимание назначения drift-мониторинга и базовой логики реагирования на деградацию модели.

## Проверяемые результаты обучения

- различает data drift и concept drift;
- понимает смысл сравнения текущих и референсных данных;
- интерпретирует сигналы деградации модели;
- связывает drift-сигнал с возможными действиями сопровождения.

## Инструкция студенту

В каждом задании выберите один правильный ответ из пяти предложенных вариантов. При решении обращайте внимание на различие между изменением данных, изменением зависимости между данными и целевой переменной, а также техническими сбоями сервиса.

## Задания

### Вопрос 1
Что наиболее точно описывает data drift?

- A. Изменение распределения входных данных относительно референсного периода
- B. Рост количества pod'ов в Kubernetes
- C. Изменение бизнес-требований к продукту
- D. Рост сетевой задержки
- E. Замена библиотеки визуализации

### Вопрос 2
Что наиболее точно описывает concept drift?

- A. Изменение формата логов приложения
- B. Изменение зависимости между входными данными и целевой переменной
- C. Снижение числа наблюдений в выборке
- D. Переход от batch к online inference
- E. Рост пользовательской нагрузки

### Вопрос 3
Какова главная цель drift-мониторинга?

- A. Заменить процесс обучения модели
- B. Автоматически удалять неиспользуемые признаки
- C. Раннее обнаружение изменений, влияющих на качество модели
- D. Повысить скорость работы Docker daemon
- E. Снизить объём логов

### Вопрос 4
Какой сигнал с наибольшей вероятностью указывает на concept drift, а не только на технический сбой сервиса?

- A. Кратковременный рост времени ответа API при стабильном качестве модели
- B. Увеличение доли 5xx-ошибок из-за ошибки конфигурации
- C. Систематическое снижение бизнес-метрики качества при стабильных инфраструктурных метриках
- D. Кратковременное падение кластера Kubernetes
- E. Рост использования диска логами

### Вопрос 5
Зачем используется референсный датасет в drift-мониторинге?

- A. Для хранения секретов доступа
- B. Для сравнения текущих данных с эталонным распределением
- C. Для публикации модели в registry
- D. Для настройки CI/CD pipeline
- E. Для описания ролей команды

### Вопрос 6
Какое действие наиболее обоснованно при устойчивом drift-сигнале и подтверждённой деградации качества модели?

- A. Игнорировать сигнал, если сервис отвечает без ошибок
- B. Удалить логи за текущий период
- C. Проанализировать причины, проверить данные и рассмотреть переобучение или пересмотр модели
- D. Немедленно отключить сервис навсегда
- E. Отключить сбор метрик

### Вопрос 7
Какой из следующих признаков чаще всего свидетельствует о data drift, а не о concept drift?

- A. Изменение распределения входных признаков без заметного изменения отношения «признаки–цель»
- B. Изменение бизнес-логики таргета при неизменных данных
- C. Массовый отказ узлов кластера
- D. Падение latency API на короткий период
- E. Изменение формата ответа API

### Вопрос 8
Какой подход к порогам алертинга в drift-мониторинге наиболее корректен?

- A. Задавать пороги случайным образом
- B. Использовать только одно универсальное пороговое значение для всех признаков
- C. Настраивать пороги, исходя из статистической значимости изменений и бизнес-рисков
- D. Всегда использовать самые низкие пороги, чтобы получать больше алертов
- E. Не использовать пороги вообще

### Вопрос 9
Какой инструмент чаще всего применяют для визуализации результатов drift-анализа и отчётов по качеству данных?

- A. Evidently
- B. Docker Hub
- C. Git LFS
- D. MinIO
- E. Terraform

### Вопрос 10
Какое из утверждений о связи drift-сигналов и переобучения модели наиболее корректно?

- A. Любой drift-сигнал требует немедленного переобучения модели
- B. Drift-сигналы не имеют отношения к решению о переобучении
- C. Drift-сигналы являются одним из оснований для анализа необходимости переобучения и пересмотра признаков
- D. Drift-сигналы важны только на этапе обучения, а не в продуктиве
- E. Drift-сигналы можно игнорировать, если нет жалоб пользователей
