# Тест 8.1. Безопасность ML-платформ и governance моделей

## Цель

Проверить понимание основных принципов безопасности ML-платформ и governance моделей, управления секретами (Vault, GitHub Secrets), RBAC, model cards, datasheets, процедур аудита, регуляторных требований.

## Инструкция

Выберите один правильный ответ в каждом задании.

## Задания

1. Для чего в ML-платформе чаще всего используют HashiCorp Vault или аналогичные secret management-системы?<br><br>
   а) Для безопасного хранения, выдачи и ротации секретов, например API-ключей, токенов и паролей.<br><br>
   б) Для автоматического увеличения accuracy модели.<br><br>
   в) Для хранения обучающих датасетов вместо object storage.<br><br>
   г) Для визуализации метрик качества модели.<br><br>
   д) Для замены системы контроля версий Git.<br><br>

2. Какой подход к хранению секретов в репозитории является наиболее безопасным?<br><br>
   а) Хранить секреты прямо в config.py, если репозиторий приватный.<br><br>
   б) Коммитить .env-файл, чтобы все участники команды имели одинаковые настройки.<br><br>
   в) Хранить секреты в GitHub Secrets, Vault или аналогичном хранилище, а в коде использовать ссылки на переменные окружения.<br><br>
   г) Публиковать секреты в README, но ограничить доступ к репозиторию.<br><br>
   д) Передавать секреты в сообщениях pull request для удобства ревью.<br><br>

3. Что такое RBAC в контексте ML-платформы?<br><br>
   а) Механизм управления доступом на основе ролей пользователей и сервисных аккаунтов.<br><br>
   б) Алгоритм балансировки нагрузки между inference-сервисами.<br><br>
   в) Формат сериализации моделей TensorFlow.<br><br>
   г) Метод автоматического подбора гиперпараметров.<br><br>
   д) Тип метрики для оценки fairness модели.<br><br>

4. Какой принцип наиболее важен при настройке RBAC для ML-платформы?<br><br>
   а) Всем пользователям лучше сразу выдавать права администратора, чтобы не блокировать работу.<br><br>
   б) Доступы следует выдавать по принципу минимально необходимых привилегий.<br><br>
   в) RBAC нужен только для production, но не нужен для staging.<br><br>
   г) Один общий service account для всех пайплайнов упрощает аудит и повышает безопасность.<br><br>
   д) Права доступа не нужно пересматривать после изменения ролей сотрудников.<br><br>

5. Что лучше всего описывает назначение model card?<br><br>
   а) Документ, описывающий модель, её назначение, ограничения, данные, метрики, риски и условия применимости.<br><br>
   б) Файл с весами модели в бинарном формате.<br><br>
   в) Kubernetes-манифест для запуска inference-сервиса.<br><br>
   г) Секретный токен для доступа к model registry.<br><br>
   д) Скрипт для автоматической балансировки нагрузки.<br><br>

6. Для чего используются datasheets for datasets?<br><br>
   а) Для замены train/test split.<br><br>
   б) Для хранения паролей к базе данных.<br><br>
   в) Для документирования происхождения, состава, способов сбора, ограничений и потенциальных рисков датасета.<br><br>
   г) Для автоматической компиляции Docker-образа.<br><br>
   д) Для ускорения GPU-инференса.<br><br>

7. Что должно быть частью базовой процедуры аудита ML-модели перед выводом в production?<br><br>
   а) Только проверка того, что notebook запускается без ошибок на компьютере разработчика.<br><br>
   б) Только сравнение размера Docker-образа с предыдущей версией.<br><br>
   в) Удаление всех логов обучения для экономии места.<br><br>
   г) Проверка источников данных, версий модели, метрик качества, ограничений, владельцев, approvals и истории изменений.<br><br>
   д) Отключение мониторинга после успешного деплоя.<br><br>

8. Почему небезопасно использовать один общий production-токен для всех ML-пайплайнов и пользователей?<br><br>
   а) Потому что это усложняет аудит действий, повышает риск утечки и нарушает принцип минимальных привилегий.<br><br>
   б) Потому что общий токен всегда снижает latency inference-сервиса.<br><br>
   в) Потому что GitHub Secrets не поддерживает токены.<br><br>
   г) Потому что RBAC работает только без токенов.<br><br>
   д) Потому что общий токен невозможно использовать в CI/CD.<br><br>

9. Что является примером корректного использования GitHub Secrets в CI/CD для ML-проекта?<br><br>
   а) Добавить токен доступа к MLflow в исходный код, чтобы workflow был воспроизводимым.<br><br>
   б) Сохранить токен доступа к MLflow Tracking Server в GitHub Secrets и использовать его в workflow через переменные окружения.<br><br>
   в) Записать токен в комментарий к pull request.<br><br>
   г) Сохранить токен в публичном Dockerfile.<br><br>
   д) Отключить секреты и использовать только hardcoded credentials.<br><br>

10. Какой артефакт или процесс лучше всего помогает подтвердить соответствие модели governance- и регуляторным требованиям?<br><br>
   а) Только высокая accuracy на обучающей выборке.<br><br>
   б) Только наличие FastAPI endpoint /predict.<br><br>
   в) Только использование GPU при обучении.<br><br>
   г) Только приватный репозиторий без документации.<br><br>
   д) Документированная цепочка происхождения данных и модели, approvals, model card, datasheet, логи экспериментов и история деплоев.<br><br>
