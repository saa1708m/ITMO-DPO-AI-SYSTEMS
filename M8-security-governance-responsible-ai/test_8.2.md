# Тест 8.2. Ответственный ИИ и устойчивость моделей

## Цель

Проверить понимание принципов ответственного ИИ и устойчивости моделей, fairness, bias, Fairlearn, AIF360, adversarial attacks, объяснимости (SHAP, LIME), ГОСТ Р 71476-2024.

## Инструкция

Выберите один правильный ответ в каждом задании.

## Задания

1. Что наиболее точно описывает bias в контексте ответственного ИИ?<br><br>
   а) Случайное изменение значения loss во время обучения модели.<br><br>
   б) Систематическое смещение или необъективность модели, из-за которой разные группы пользователей могут получать неравные результаты.<br><br>
   в) Увеличение throughput inference-сервиса при росте числа реплик.<br><br>
   г) Ошибка сериализации модели при сохранении в model registry.<br><br>
   д) Любое снижение accuracy модели на тестовой выборке.<br><br>

2. Какой инструмент наиболее уместен для оценки fairness-метрик и анализа различий качества модели между группами пользователей?<br><br>
   а) Docker Compose.<br><br>
   б) Fairlearn.<br><br>
   в) Terraform.<br><br>
   г) Prometheus Node Exporter.<br><br>
   д) Gunicorn.<br><br>

3. Какой пример лучше всего соответствует проверке fairness модели кредитного скоринга?<br><br>
   а) Проверить только общий ROC-AUC на всей тестовой выборке без разбиения на группы.<br><br>
   б) Проверить, что Docker-образ модели меньше 1 ГБ.<br><br>
   в) Сравнить false positive rate, false negative rate и approval rate между релевантными группами пользователей.<br><br>
   г) Проверить, что модель обучалась не дольше одного часа.<br><br>
   д) Проверить только наличие REST endpoint /predict.<br><br>

4. Для чего в governance-процессе используется model card?<br><br>
   а) Для хранения production-секретов модели.<br><br>
   б) Для документирования назначения модели, данных, метрик, ограничений, рисков, fairness-проверок и условий допустимого применения.<br><br>
   в) Для автоматической балансировки нагрузки между версиями модели.<br><br>
   г) Для замены RBAC в ML-платформе.<br><br>
   д) Для шифрования датасета перед обучением.<br><br>

5. Какое содержание наиболее уместно включить в model card для модели, применяемой в чувствительном домене?<br><br>
   а) Назначение модели, описание данных, метрики качества, fairness-метрики, ограничения, known failure modes, процедуру мониторинга и ответственных владельцев.<br><br>
   б) Только имя автора модели и дату последнего коммита.<br><br>
   в) Только список Python-библиотек, установленных в окружении.<br><br>
   г) Только ссылку на Dockerfile.<br><br>
   д) Только значение accuracy на обучающей выборке.<br><br>

6. Что лучше всего описывает adversarial attack на ML-модель?<br><br>
   а) Плановое переобучение модели на новых данных.<br><br>
   б) Автоматическое масштабирование inference-сервиса при росте CPU.<br><br>
   в) Целенаправленное воздействие на входные данные или окружение модели, чтобы вызвать ошибочное или нежелательное предсказание.<br><br>
   г) Сравнение двух версий модели в A/B-тесте.<br><br>
   д) Документирование ограничений модели в model card.<br><br>

7. Какой подход наиболее уместен для повышения устойчивости модели к adversarial-примерам?<br><br>
   а) Игнорировать выбросы и не логировать подозрительные запросы.<br><br>
   б) Использовать adversarial testing, валидацию входных данных, мониторинг аномалий и при необходимости adversarial training.<br><br>
   в) Выдать всем пользователям права администратора, чтобы упростить отладку.<br><br>
   г) Удалить тестовую выборку после обучения.<br><br>
   д) Отключить контроль версий модели.<br><br>

8. Чем SHAP и LIME полезны в аудите ML-моделей?<br><br>
   а) Они автоматически устраняют все bias-проблемы в данных.<br><br>
   б) Они заменяют необходимость model card.<br><br>
   в) Они используются только для хранения секретов.<br><br>
   г) Они гарантируют соответствие любым регуляторным требованиям без дополнительной проверки.<br><br>
   д) Они позволяют интерпретировать вклад признаков в предсказания модели и выявлять потенциально нежелательные зависимости.<br><br>

9. Какой набор действий лучше всего соответствует полноценному аудиту модели в рамках governance ML-платформы?<br><br>
   а) Проверить только, что модель запускается локально у разработчика.<br><br>
   б) Проверить только latency REST API без анализа качества модели.<br><br>
   в) Проверить происхождение данных, версию модели, model card, fairness-метрики, explainability-отчёты, права доступа, approvals, логи экспериментов и историю деплоев.<br><br>
   г) Проверить только размер Docker-образа и количество pod’ов в Kubernetes.<br><br>
   д) Проверить только, что модель сохранена в формате pickle.<br><br>

10. Какое утверждение наиболее корректно связывает ГОСТ Р 71476-2024 и governance ML-систем?<br><br>
   а) ГОСТ Р 71476-2024 автоматически сертифицирует любую ML-модель как безопасную.<br><br>
   б) ГОСТ Р 71476-2024 задаёт терминологическую и концептуальную основу, которую можно использовать для единообразного описания понятий ИИ, включая объяснимость, предвзятость и устойчивость, в документации и аудитах.<br><br>
   в) ГОСТ Р 71476-2024 заменяет необходимость RBAC и управления доступами.<br><br>
   г) ГОСТ Р 71476-2024 требует использовать только Fairlearn и запрещает AIF360.<br><br>
   д) ГОСТ Р 71476-2024 применяется только к компьютерному зрению и не относится к другим ML-системам.<br><br>
