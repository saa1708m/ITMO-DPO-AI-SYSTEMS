# Кейс №2. Система интеллектуального анализа и проверки документов

## Паспорт КИМ

| Параметр | Значение |
|----------|----------|
| **Вид КИМ** | Курсовой проект (кейс) |
| **Модули** | 1, 2, 3, 5, 6, 8 |
| **Компетенции** | LC-2, LC-3, LC-5 |
| **Индикаторы** | LC-2.1–2.3, LC-3.1, LC-5.1–5.3, LC-5.5 |
| **Уровень** | С (средний) |
| **Режим выполнения** | Индивидуально или в паре |
| **Срок выполнения** | 3 недели |
| **Максимальный балл** | 10 |

## Назначение

Разработать систему автоматической проверки и анализа документов, способную извлекать ключевую информацию, проверять соответствие шаблонам и выявлять расхождения. Проект основан на реальных решениях Т1: «Сайбокс.Док» и ИИ-сканер для распознавания документов. По данным Т1, внедрение такого решения может обеспечить экономию до 80% времени на проверку тендерной документации.

## Проверяемые результаты обучения

Студент должен уметь:

- организовывать пайплайн обработки неструктурированных документов;
- реализовывать модели классификации и извлечения сущностей (NER);
- строить ETL-пайплайны для документов;
- развёртывать API с управлением доступом;
- обеспечивать безопасность и аудит обработки документов.

## Постановка задания

### 1. Подготовка данных (Модуль 2)

**Задание:**
- Соберите коллекцию документов (PDF, DOCX) с разметкой (например, договоры, счета, акты).
- Выполните извлечение текста (используйте OCR при необходимости, например, Tesseract).
- Проведите EDA: структура документов, частотность терминов.
- Настройте версионирование данных через DVC.

**Ожидаемый результат:**
- Корпус документов в `data/raw/`.
- Извлечённые тексты в `data/processed/`.
- DVC-репозиторий.

### 2. Разработка моделей (Модуль 3)

**Задание:**
- Реализуйте модель для классификации типа документа (например, договор, счёт, акт).
- Реализуйте модель для извлечения именованных сущностей (NER): даты, суммы, стороны, номера.
- Обучите модели, логируйте эксперименты в MLflow.
- Зарегистрируйте лучшие модели в Model Registry.

**Ожидаемый результат:**
- Код обучения моделей.
- MLflow эксперименты.
- Зарегистрированные модели.

### 3. Построение пайплайна проверки (Модуль 5)

**Задание:**
- Разработайте ETL-пайплайн: загрузка документа → распознавание → извлечение сущностей → проверка соответствия шаблону.
- Настройте Airflow DAG для автоматической обработки входящих документов.

**Ожидаемый результат:**
- Код ETL-пайплайна.
- Airflow DAG в `dags/document_processing.py`.

### 4. Развёртывание API (Модуль 6)

**Задание:**
- Создайте REST API (FastAPI) для загрузки документа и получения результатов анализа.
- Обеспечьте горизонтальное масштабирование (Kubernetes).

**Ожидаемый результат:**
- Работающий API с эндпоинтами `/upload` и `/status/{task_id}`.
- Kubernetes конфигурация.

### 5. Безопасность и Governance (Модуль 8)

**Задание:**
- Настройте управление доступом к документам (RBAC).
- Создайте model card для каждой модели.
- Обеспечьте логирование всех действий для аудита.

**Ожидаемый результат:**
- Model cards для каждой модели.
- Логирование в ELK стеке.

## Формат сдачи

- Git-репозиторий с кодом и конфигами.
- Демонстрация работы API на тестовых документах.
- Отчёт с описанием архитектуры и результатов.

## Критерии оценивания

| Критерий | 0 баллов | 1 балл | 2 балла |
|----------|----------|--------|---------|
| **Обработка документов** | Не реализована | Базовая обработка (извлечение текста) | Полный пайплайн: извлечение, NER, классификация |
| **Качество моделей** | Модели не обучены | Модели есть, но качество низкое | Высокое качество, обоснованный выбор метрик |
| **ETL и оркестрация** | Отсутствует | Частичная реализация | Полный ETL + Airflow DAG |
| **API и безопасность** | Нет API | API есть, но без безопасности | Полноценный API с RBAC и логированием |
| **Документация** | Отсутствует | Базовая | Полная документация + model cards |
