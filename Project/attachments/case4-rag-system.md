# Кейс №4. RAG-система для работы с корпоративной документацией

## Паспорт КИМ

| Параметр | Значение |
|----------|----------|
| **Вид КИМ** | Курсовой проект (кейс) |
| **Модули** | 2, 3, 4, 5, 6, 8 |
| **Компетенции** | LC-2, LC-3, LC-5 |
| **Индикаторы** | LC-2.1, LC-3.1, LC-5.1–5.3, LC-5.5 |
| **Уровень** | С–П (средний–продвинутый) |
| **Режим выполнения** | Индивидуально |
| **Срок выполнения** | 3 недели |
| **Максимальный балл** | 10 |

## Назначение

Разработать RAG-систему (Retrieval-Augmented Generation) для ответов на вопросы по корпоративной документации. Проект использует подход, аналогичный реализованному в платформе «Сайбокс» для ВЭБ.РФ, где каждый сотрудник может самостоятельно создавать ИИ-помощников для оптимизации рабочих процессов.

## Проверяемые результаты обучения

Студент должен уметь:

- подготавливать корпус документов для RAG;
- строить пайплайн индексации и поиска;
- интегрировать LLM для генерации ответов;
- контейнеризировать RAG-систему;
- обеспечивать безопасность и ограничения использования.

## Постановка задания

### 1. Подготовка корпуса документов (Модуль 2)

**Задание:**
- Соберите коллекцию документов (PDF, DOCX, TXT) по выбранной тематике.
- Очистите тексты, выполните чанкинг (разбиение на фрагменты).
- Настройте DVC для версионирования данных.

**Ожидаемый результат:**
- Корпус документов в `data/raw/`.
- Чанки в `data/chunks/`.
- DVC-репозиторий.

### 2. Разработка RAG-пайплайна (Модуль 3)

**Задание:**
- Выберите и настройте эмбеддинг-модель (например, sentence-transformers).
- Постройте векторную базу данных (FAISS, Chroma, Qdrant).
- Настройте LLM для генерации ответов (можно использовать открытые модели или API).

**Ожидаемый результат:**
- Код построения векторной базы.
- Скрипт для генерации ответов.

### 3. Контейнеризация (Модуль 4)

**Задание:**
- Создайте Docker-образы для всех компонентов RAG-системы.
- Настройте docker-compose.

**Ожидаемый результат:**
- Dockerfile и docker-compose.yml.

### 4. Оркестрация и CI/CD (Модуль 5)

**Задание:**
- Реализуйте пайплайн для обновления векторной базы при добавлении новых документов.
- Настройте CI/CD для автоматического деплоя.

**Ожидаемый результат:**
- Airflow DAG или скрипт обновления.
- CI/CD конфигурация.

### 5. Развёртывание API (Модуль 6)

**Задание:**
- Создайте REST API (FastAPI) для приёма вопросов и возврата ответов с цитированием источников.

**Ожидаемый результат:**
- Работающий REST API с эндпоинтами `/ask` и `/health`.

### 6. Безопасность (Модуль 8)

**Задание:**
- Настройте управление секретами.
- Создайте документацию по ограничениям системы.

**Ожидаемый результат:**
- Файл `LIMITATIONS.md` с описанием ограничений.

## Формат сдачи

- Git-репозиторий с кодом.
- Демонстрация работы API.
- Отчёт с описанием архитектуры и результатов тестирования.

## Критерии оценивания

| Критерий | 0 баллов | 1 балл | 2 балла |
|----------|----------|--------|---------|
| **Подготовка данных** | Не выполнена | Базовая очистка | Полный пайплайн: очистка, чанкинг, эмбеддинги |
| **RAG-пайплайн** | Отсутствует | Частичная реализация | Полноценный RAG с ретривером и генерацией |
| **Контейнеризация** | Отсутствует | Частичная | Полный Docker-стек |
| **API и документация** | Нет API | API есть, но без документации | Полноценный API + документация |
| **Безопасность** | Отсутствует | Частичная | Полная: секреты, ограничения, model card |
